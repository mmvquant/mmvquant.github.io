Building Production RAG Systems: Lessons from Deploying LLMs at Scale

Dec 15, 2025 12 min read 4,832 views

Retrieval-Augmented Generation (RAG) has become the de facto pattern for building LLM applications that need to reference private or up-to-date knowledge. But moving from a notebook prototype to a production system is where most teams stumble.

01. The Architecture

In production RAG, the retrieval quality matters more than the model size. We found that a well-tuned retrieval pipeline with a smaller model consistently outperformed a larger model with naive retrieval.

rag_pipeline.py 
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA
from transformers import AutoTokenizer, AutoModelForCausalLM

class ProductionRAG:
    def __init__(self, config: dict):
        self.embeddings = HuggingFaceEmbeddings(
            model_name="BAAI/bge-large-en-v1.5",
            encode_kwargs={"normalize_embeddings": True}
        )
        self.vectorstore = Chroma(
            persist_directory=config["chroma_path"],
            embedding_function=self.embeddings
        )

    def query(self, question: str, k=5) -> str:
        retriever = self.vectorstore.as_retriever(
            search_type="mmr",
            search_kwargs={"k": k, "fetch_k": 20}
        )
        docs = retriever.invoke(question)
        return self._generate(question, docs)

02. Vector Database Selection

We benchmarked four vector databases across three dimensions that matter in production: latency at scale, filtering capability, and operational complexity. Here are the results:

Database P99 Latency Filtering Ops Score
Qdrant 12ms Excellent 9/10
Weaviate 18ms Excellent 7/10
Pinecone 8ms Good 8/10
Chroma 25ms Basic 6/10

Key Insight: For most teams starting out, Chroma is perfectly fine. But if you need metadata filtering at scale (which you will), invest time in Qdrant early. The migration cost only increases.

03. Chunking Strategies

The biggest impact on retrieval quality isn't the embedding model — it's how you chunk your documents. After months of experimentation, here's what actually works:

chunking.py 
def semantic_chunk(text: str, threshold=0.65):
    sentences = sent_tokenize(text)
    embeddings = model.encode(sentences)
    chunks, current = [], [sentences[0]]

    for i in range(1, len(sentences)):
        sim = cosine_similarity(
            embeddings[i-1], embeddings[i]
        )
        if sim < threshold:
            chunks.append(" ".join(current))
            current = [sentences[i]]
        else:
            current.append(sentences[i])

    chunks.append(" ".join(current))
    return chunks  # ~35% better retrieval
